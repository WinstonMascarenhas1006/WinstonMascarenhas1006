## Research & professional interests

Cybersecurity is not a side topic in my work — it is the **central through-line**. I am particularly interested in:

1. **Defensive tooling & detection** — turning noisy signals (logs, traffic, persistence artifacts) into actionable findings  
2. **Adversary-informed practice** — understanding offensive techniques in controlled labs so defenses are grounded, not generic  
3. **Hardening & assurance** — configuration auditing, exposure reduction, and repeatable verification  
4. **Secure software construction** — applying security thinking to full-stack systems (authn/z boundaries, abuse cases, operational safety)

My broader computing background (software process, web systems, applied engineering) supports this agenda: I treat security as something that must be **implemented, tested, and explained**, not only discussed.

---

## Why cybersecurity

I am drawn to cybersecurity because it combines rigorous systems thinking with high-stakes problem solving. Effective security work demands:

- precise models of trust and failure  
- disciplined experimentation (especially in isolated labs)  
- clear communication of risk and remediation  

That combination — analytical depth, practical construction, and responsible disclosure/lab ethics — is what I want to keep deepening through projects, study, and professional practice.

## Method

I try to keep security work **legible and accountable**:

- Prefer reproducible labs and explicit scope (what is in-bounds / out-of-bounds)  
- Document assumptions, limitations, and ethical constraints  
- Prefer measurable outputs (reports, findings, structured exports) over opaque “magic”  
- Use software engineering discipline (version control, reviews, iterative refinement) so security tools remain maintainable
